# 主业务流程重分析与重设计（2026-04-25）

## 1. 结论摘要

1. 当前实现确实把本质上“工序顺序推进”的业务拆成了多个并行常驻线程执行。
2. 这种拆分在设备稳定时可以跑通，但在链路抖动或重连阶段，会放大竞态和状态不一致风险。
3. 你提出的方向是对的：等待 PLC 地址值变化完全可以用异步方式实现，而且更适合做“顺序清晰”的编排。
4. 建议采用“单一流程编排器 + 异步等待 + 显式状态机 + 串行PLC访问”的设计，保留并发能力但避免多线程互相踩状态。

## 2. 现状分析（代码证据）

### 2.1 启动阶段一次性拉起多个长期任务

在 [Inkjet_Print_View/FrmMain.cs](../Inkjet_Print_View/FrmMain.cs#L247) 到 [Inkjet_Print_View/FrmMain.cs](../Inkjet_Print_View/FrmMain.cs#L255) 这段，系统同时启动了清洗、称重1、称重2、冷喷、监控、报警、温度写入等多个 LongRunning 任务。

这意味着：
- 每个任务都在独立循环里轮询 PLC。
- 它们共享同一个业务对象和大量共享状态。
- 业务顺序约束被分散到多个线程中的 if/while 逻辑里，而不是单一流程定义里。

## 3. 运行日志分析（你提供的 2026-04-21）

本次日志最突出的事实不是工序逻辑错误，而是“长时间连接失败占据了绝大部分运行时间”。

### 3.1 关键证据

- 启动与连接开始：第 1 行、第 5 行。
- 紧接着出现连续的 PLC 连接失败：第 6 行开始高频重复。
- 心跳写失败反复出现：例如第 44 行、第 85 行。
- 套接字状态异常（非正常 socket 对象）：例如第 212、437、656、1074、1200 行等。
- 以 PLC 地址字符串为关键词的异常相关行高达 4241 行（同一份日志内统计）。

### 3.2 对分析结论的影响

因为日志中大量时间都在连不上 PLC，业务阶段日志覆盖不足，所以不能仅凭这一天的日志去断言“某个工序必然有严格业务 bug”。

但可以确认两点：
1. 当前架构在链路抖动场景下可观测性不足，难以从日志还原单条码全链路。
2. 多线程轮询 + 共享连接在高故障期会使问题定位成本急剧上升。

## 4. 对你问题的直接回答

### 4.1 这个上位机整体设计是否有问题？

有改进空间，主要是架构层面的可维护性与抗抖动能力问题，不是单点语法问题。

### 4.2 是否把顺序流程拆成了几个线程？

是。当前是“工序线程并行 + 数据库汇合”。在设备稳定时可运行，但“流程定义”不直观，也不利于故障期稳定。

### 4.3 等待 PLC 地址值是否可以异步实现？

可以，而且建议这么做。

推荐把“等待地址变化”封装为统一异步原语，例如：
- WaitAddressAsync(address, expected, timeout, cancellation)
- WaitAddressChangeAsync(address, from, to, timeout, cancellation)

这样可替代多处 while(true)+Sleep，并统一超时、取消、日志和重试策略。

## 5. 重设计方案（顺序清晰版）

## 5.1 目标

让代码结构直接映射 PLC 工序顺序：
清洗 -> 称重1 -> 冷喷 -> 监控 -> 称重2 -> 复位/收尾

## 5.2 核心设计

### A. 单一流程编排器（ProcessOrchestrator）

职责：
- 每次只负责一个流程实例（按条码或工件ID）。
- 串行执行各工序步骤。
- 每步使用统一异步等待与统一写入确认。

### B. PLC I/O 网关（PlcGateway）

职责：
- 所有 PLC 读写统一从这里走。
- 内部串行化写操作，避免跨线程同时写。
- 提供读回确认：WriteThenConfirmAsync(address, value, confirmReadAddress, expected, timeout)

### C. 工序状态机（ProcessState）

建议状态：
- Idle
- WaitingPlacementReady
- WaitingWeight1Ready
- WaitingColdSprayReady
- WaitingMonitorReady
- WaitingWeight2Ready
- Finishing
- Completed
- Faulted

### D. 观察性上下文（FlowContext）

每个流程实例都带：
- FlowId
- Code
- CurrentState
- StartedAt
- RetryCount
- PlcRoundTripMs

日志统一输出这些字段，保证可追踪。

## 5.3 执行示意（伪流程）

1. Wait D5000==1，读取条码，建立 FlowContext。
2. Wait D5500==1，执行称重1，写 D5700=2 并确认。
3. Wait D5670==1，执行冷喷，Wait D5670==2，写 D5710=2 并确认。
4. Wait D5671==1，执行监控，Wait D5671==2，写 D5711=2 并确认。
5. Wait D5699==1，执行称重2，写 D5714=2 并确认。
6. 统一收尾、落库、输出结果。

## 5.4 错误处理统一策略

- 每步都必须有 timeout。
- 每个 PLC 写入都要“写入 + 读回确认 + 有限重试”。
- 连接恢复由统一连接管理器处理，业务步骤不直接到处触发重连。
- 超过最大重试进入 Faulted，记录完整上下文并可人工重放。

## 6. 分阶段迁移步骤（建议）

### Phase 1：先稳连接与可观测性（最小改动）

1. 统一日志字段：FlowId、Code、State、Address、Value、Retry、ElapsedMs。
2. 把所有关键写入改成写后确认。
3. 重连逻辑集中管理，避免业务线程各自重连。

### Phase 2：替换轮询等待为异步等待原语

1. 引入 WaitAddressAsync。
2. 将各工序内部的 while(true)+Sleep 逐段替换。
3. 保持原线程结构先不动，先把等待方式一致化。

### Phase 3：引入单一编排器与状态机

1. 新建 ProcessOrchestrator，先接入一条完整链路。
2. 旧线程与新编排器灰度并行（开关控制）。
3. 验证稳定后逐步下线旧线程任务。

## 7. 对 D5710 问题的关联结论

在“高连接失败”背景下，D5710 异常观测（一直 0 或瞬时 0）可能来自两类情况并存：

1. 写入未达 PLC：连接抖动/写入时 socket 异常。
2. 写入已达但被 PLC 快速消费后清零：如果只看慢速监控画面，会表现为“没写上”。

因此必须依赖“写入返回 + 读回确认 + 带 FlowId 日志”三联证据。
